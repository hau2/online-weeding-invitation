<objective>
Build the invitation editor page with side-by-side form and live preview layout. The form uses collapsible accordion sections for all invitation fields, auto-saves via 800ms debounced PATCH, and the preview pane renders the actual template component in a phone mockup frame that updates in real time as the user types.

Purpose: This is the core editing experience. Couples spend most of their time here filling in details and seeing how their invitation looks. The preview must update instantly (from local state, not network), and saving must be invisible (no save button).

Output: Complete editor page with form, preview, template selector, and auto-save hook. Wave 0 test stubs for all frontend components in this phase.

    1. Create `useAutoSave.ts` -- custom hook for 800ms debounced PATCH:
       - Export type `SaveStatus = 'idle' | 'saving' | 'saved' | 'error'`
       - Hook signature: `useAutoSave(invitationId: string, delay?: number)` returns `{ save: (data: Partial<Invitation>) => void, status: SaveStatus }`
       - Use `useRef<ReturnType<typeof setTimeout>>` for timer and `useRef<Partial<Invitation>>` for latest data (prevents stale closure -- Research Pitfall #2)
       - On `save()` call: update latestDataRef, clear existing timer, set new timer for `delay` ms
       - In timer callback: read from latestDataRef (not closure), set status='saving', call `apiFetch<Invitation>('/invitations/${invitationId}', { method: 'PATCH', body: latestDataRef.current, credentials: 'include' })`, set status='saved' on success or 'error' on failure
       - On error: call `toast.error()` via sonner with the error message
       - Cleanup on unmount: clear timer in useEffect return
       - After status becomes 'saved', auto-reset to 'idle' after 2 seconds (so the indicator fades)

    2. Create `page.tsx` -- server component that fetches invitation data:
       - Import `cookies` from 'next/headers'
       - Read the 'auth-token' cookie
       - Fetch `GET /invitations/${params.id}` from the API with the cookie forwarded as an Authorization header OR as a Cookie header (match the pattern used by the dashboard page -- use `credentials: 'include'` and forward the cookie header)
       - Use `cache: 'no-store'` to prevent stale data after auto-save (Research Pitfall #7)
       - If fetch fails or invitation not found, call `notFound()` from next/navigation
       - Render `<EditorShell invitation={data} />` with the fetched data
       - Page should be minimal -- just data fetching and component rendering

    3. After implementation, convert the useAutoSave test stubs from `it.todo()` to real tests:

       - Test: calling save() then advancing 800ms triggers apiFetch PATCH
       - Test: calling save() multiple times within 800ms only triggers one PATCH
       - Test: status transitions idle -> saving -> saved -> idle
       - Test: status goes to 'error' when apiFetch returns error
       - Test: unmount clears pending timer
