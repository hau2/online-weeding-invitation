# Phase 9.1: Public Page Redesign - Modern Full-Width Templates - Research

**Researched:** 2026-03-16
**Domain:** React component architecture, CSS/Tailwind theme system, full-width responsive layout
**Confidence:** HIGH

## Summary

This phase replaces the existing narrow 420px card-style invitation templates with 6 modern full-width section-based templates matching Stitch AI reference designs. The core architectural change is moving from 3 separate template components (TemplateTraditional, TemplateModern, TemplateMinimalist) to a single shared layout component driven by a theme configuration object. The Stitch reference HTML files (screen1.html for Modern Red, screen6.html for B&W Minimalist) provide exact Tailwind class patterns, section structures, and responsive breakpoints to replicate.

The existing codebase has clean separation points: `InvitationShell.tsx` orchestrates envelope-to-revealed flow, `TemplateRenderer.tsx` routes to template components, and `TemplateProps` interface receives the full `Invitation` object. The DesktopFrame component (botanical corners at max-w-480px) must be removed entirely -- the new full-width templates ARE the desktop experience. The `TemplateId` type union in `@repo/types` must expand from 3 to 6 values, and the `CountdownTimer` component needs rewriting from flip-card style to the Stitch card-box style.

**Primary recommendation:** Build one `SharedTemplate` component with 8 section sub-components (Hero, StickyNav, Countdown, Timeline, Gallery, Venue, BankQR, Footer), each consuming a `ThemeConfig` object for visual customization. Define 6 hardcoded theme configs. Map old templateId values to new themes in code (no DB migration). Use Plus Jakarta Sans font for public pages as specified by Stitch designs.

### Pattern 1: Theme Config Object
**What:** A TypeScript interface that defines all visual properties a section needs, allowing one layout component to render 6+ distinct themes.
**When to use:** Always -- this is the core architecture.
**Example:**
```typescript
// Source: derived from Stitch HTML analysis (screen1.html + screen6.html)
export interface ThemeConfig {
  id: string                    // 'modern-red', 'minimalist-bw', etc.
  name: string                  // Display name: 'Hien dai - Do'
  // Colors
  primaryColor: string          // '#ec1349' | '#171717' etc.
  backgroundColor: string       // '#f8f6f6' | '#ffffff' etc.
  surfaceColor: string          // Section alternate bg
  textColor: string             // Main text color
  mutedTextColor: string        // Secondary text color
  // Typography
  headingWeight: string         // 'font-black' | 'font-light'
  bodyWeight: string            // 'font-medium' | 'font-light'
  letterSpacing: string         // 'tracking-tight' | 'tracking-widest'
  textTransform: string         // 'uppercase' | 'normal-case'
  // Borders & Shapes
  borderRadius: string          // 'rounded-xl' | 'rounded-none'
  cardBorderRadius: string      // 'rounded-2xl' | 'rounded-none'
  // Hero
  heroOverlay: string           // gradient overlay CSS
  heroMinHeight: string         // 'min-h-[560px]' | 'min-h-[600px]'
  // Gallery
  galleryEffect: string         // '' | 'grayscale hover:grayscale-0'
  galleryGap: string            // 'gap-3 md:gap-6' | 'gap-2 md:gap-4'
  // Petals
  petalColors: string[]         // Array of hex colors for FallingPetals
  petalEnabled: boolean         // false for B&W Minimalist
  // Navigation
  navStyle: 'colored' | 'mono'  // Primary colored vs monochrome
  // Footer
  footerBg: string              // 'bg-white' | 'bg-neutral-900'
  footerTextColor: string       // Text color for footer
}
```

### Pattern 3: Backward-Compatible Template ID Mapping
**What:** Map old `TemplateId` values to new theme IDs in the TemplateRenderer, with no DB migration.
**When to use:** In `TemplateRenderer.tsx` and `ThemeConfig` lookup.
**Example:**
```typescript
// themes/index.ts
export type ThemeId =
  | 'modern-red'
  | 'soft-pink'
  | 'brown-gold'
  | 'olive-green'
  | 'minimalist-bw'
  | 'classic-red-gold'

// Backward compatibility mapping
const LEGACY_MAP: Record<string, ThemeId> = {
  traditional: 'classic-red-gold',
  modern: 'modern-red',
  minimalist: 'minimalist-bw',
}

export function getTheme(templateId: string): ThemeConfig {
  const themeId = LEGACY_MAP[templateId] ?? templateId
  return THEMES[themeId as ThemeId] ?? THEMES['modern-red']
}
```

### Pattern 4: CSS Custom Properties for Runtime Theme Colors
**What:** Set theme colors as CSS custom properties on a wrapper div so Tailwind classes like `text-[var(--theme-primary)]` work.
**When to use:** For theme-specific colors that cannot be expressed as static Tailwind classes.
**Example:**
```typescript
// In SharedTemplate or section wrapper
<div
  style={{
    '--theme-primary': theme.primaryColor,
    '--theme-bg': theme.backgroundColor,
    '--theme-surface': theme.surfaceColor,
    '--theme-text': theme.textColor,
    '--theme-muted': theme.mutedTextColor,
  } as React.CSSProperties}
>
  {/* Sections can use: text-[var(--theme-primary)], bg-[var(--theme-bg)], etc. */}
</div>
```

### Anti-Patterns to Avoid
- **Separate components per theme:** Do NOT create 6 separate template components. One shared layout + theme config is the locked decision.
- **Hardcoded colors in JSX:** Use theme config properties or CSS custom properties, never hardcode hex values in section components. Only theme config files should contain hex colors.
- **Breaking the envelope flow:** The envelope animation is unchanged. Only the REVEALED stage uses the new full-width template. Do not touch EnvelopeAnimation.tsx.
- **Dark mode in public pages:** The Stitch designs show light mode only. Do NOT implement dark mode for public invitation pages (the `dark:` classes in Stitch HTML are for reference only).
- **Adding Material Symbols font:** Stitch uses Material Symbols Outlined, but the project uses Lucide React. Map icon names: `favorite` -> Heart, `event_available` -> CalendarCheck, `directions` -> Navigation, `location_on` -> MapPin, `volunteer_activism` -> Heart, `content_copy` -> Copy, `phone` -> Phone, `photo_library` -> Images, `music_note` -> Music, `menu` -> Menu.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Font loading + optimization | Manual @font-face + preload | `next/font/google` for Plus Jakarta Sans | Automatic subset, preload, zero CLS |
| Image optimization for gallery | Manual srcSet/sizes | `next/image` with fill + object-cover | Automatic WebP, responsive sizing, lazy loading |
| Smooth scroll to sections | Custom scroll handler | `scroll-behavior: smooth` on html + anchor hrefs | Browser-native, performant, zero JS |
| Countdown timer | Custom Date parsing | Existing `CountdownTimer` logic (keep calculation, restyle UI) | Timezone-aware logic already handles Vietnam UTC+7 |
| Copy to clipboard | Custom implementation | Existing `CopyAccountNumber` component | Already handles fallback for older browsers |
| Bank QR free-tier lock | Custom blur overlay | Existing `BankQrLock` component | Already implements blur + lock icon pattern |

**Key insight:** The Stitch designs use pure Tailwind utility classes extensively. Nearly every visual property (spacing, colors, typography, borders, shadows, animations) maps directly to Tailwind classes. The main engineering challenge is the theme configuration system, not the CSS itself.

## Common Pitfalls

### Pitfall 1: TemplateId Type Union Not Extended
**What goes wrong:** TypeScript errors everywhere because `TemplateId` is still `'traditional' | 'modern' | 'minimalist'` but code references `'modern-red'`, `'minimalist-bw'`, etc.
**Why it happens:** `TemplateId` is defined in `packages/types/src/invitation.ts` and used across the entire codebase (API, web, types).
**How to avoid:** Extend `TemplateId` to include all 6 new theme IDs while keeping the 3 legacy values for backward compatibility. Update in packages/types first, then propagate.
**Warning signs:** TypeScript compile errors in TemplateRenderer, InvitationShell, EditorShell, admin theme management.

### Pitfall 2: Envelope-to-Full-Width Transition Breaking Layout
**What goes wrong:** After envelope opens, the full-width template renders inside the old max-w-480px constraint, or the transition feels jarring.
**Why it happens:** `InvitationShell` currently wraps everything in `DesktopFrame` which applies `max-w-[480px]`.
**How to avoid:** Remove `DesktopFrame` import and wrapper entirely from InvitationShell. The revealed stage should render `SharedTemplate` directly in a full-width container. Use framer-motion for a smooth scale/fade transition from envelope to full-width.
**Warning signs:** Template content appears squished on desktop, or visible layout shift after envelope opens.

### Pitfall 3: Side Filtering Breaking New Section Layout
**What goes wrong:** The `?side=` parameter filtering nullifies ceremony/bank data, but new sections like VenueSection or BankQrSection don't handle null/empty gracefully.
**Why it happens:** The existing `filteredInvitation` useMemo in InvitationShell blanks out the other side's data. New sections must handle this.
**How to avoid:** Each section component must check for data presence before rendering. VenueSection and BankQrSection should gracefully show only the active side's data.
**Warning signs:** Empty sections visible, layout gaps where hidden side's data should be.

### Pitfall 4: Plus Jakarta Sans Not Loading on Public Pages
**What goes wrong:** Public pages still use Be Vietnam Pro or Geist instead of Plus Jakarta Sans as Stitch designs specify.
**Why it happens:** The app layout.tsx applies Geist font variable globally. Public page route needs its own font configuration.
**How to avoid:** Add Plus Jakarta Sans to `lib/fonts.ts` via `next/font/google`. Apply its CSS variable class specifically to the public page layout or SharedTemplate wrapper. The `font-display` (Plus Jakarta Sans) from Stitch designs maps to `font-family: 'Plus Jakarta Sans', sans-serif`.
**Warning signs:** Text on public pages looks different from Stitch reference, wrong font weight behavior.

### Pitfall 5: Gallery Images Using Background-Image Instead of Next/Image
**What goes wrong:** Gallery section uses `background-image: url()` like the Stitch HTML, missing out on Next.js image optimization.
**Why it happens:** Direct copy of Stitch HTML patterns.
**How to avoid:** Use `next/image` with `fill` prop and `object-cover` class inside a relatively-positioned container with `aspect-[3/4]`. The visual result is identical but with automatic WebP conversion and lazy loading.
**Warning signs:** Large image payloads, no WebP serving, poor mobile performance (Phase 9 already optimized for <1MB page weight).

### Pitfall 6: Sticky Nav Z-Index Conflicting with FallingPetals/MusicPlayer
**What goes wrong:** Sticky nav overlaps or gets hidden behind falling petals, or vice versa.
**Why it happens:** FallingPetals uses z-40, MusicPlayer uses z-50. Sticky nav in Stitch uses z-40.
**How to avoid:** Establish clear z-index layers: FallingPetals z-30 (lower than nav), StickyNav z-40, MusicPlayer z-50, Watermark z-[60].
**Warning signs:** Visual stacking order issues, petals appearing over nav, nav appearing over music button.

### Pitfall 7: Hydration Mismatch from Window-Dependent Rendering
**What goes wrong:** Server-rendered HTML differs from client rendering because StickyNav uses window width, or countdown shows different time.
**Why it happens:** Next.js SSR renders once on server; client hydration must match exactly.
**How to avoid:** StickyNav uses `hidden md:flex` Tailwind classes (CSS-based responsive, no JS). Countdown renders null on server (dynamic import with ssr: false). Sections that need window/time use client-only patterns already established.
**Warning signs:** React hydration warnings in console, flash of incorrect content.

### Pitfall 8: Admin Theme Metadata Out of Sync
**What goes wrong:** Admin panel shows 3 old themes, but public pages have 6 new themes. Theme enable/disable doesn't work for new themes.
**Why it happens:** Phase 8 admin stores theme metadata in system_settings JSONB with only 3 entries.
**How to avoid:** Update the initial theme metadata in the admin system to include all 6 new theme IDs. Ensure the admin theme management CRUD operations work with new theme IDs. The admin only manages metadata (name, tag, enabled/disabled, thumbnail) -- not the actual rendering config.
**Warning signs:** Admin theme list incomplete, toggling themes has no effect on new themes.

## Transition Strategy: Envelope to Full-Width

The envelope animation currently renders inside `DesktopFrame` (max-w-480px). After opening, the revealed content also renders inside DesktopFrame. The new behavior:

1. **Envelope stage:** Render envelope animation in a centered container (no DesktopFrame). The existing ~280px envelope size is fine. FallingPetals runs at z-30.
2. **Transition:** When `envelopeOpened` becomes true, use framer-motion to animate the content wrapper from centered to full-width. A simple fade-in-up (existing pattern) works well.
3. **Revealed stage:** Render `SharedTemplate` in a full-width wrapper. No max-width constraint on the outer container. Individual sections manage their own max-width (max-w-[1200px] for hero/gallery, max-w-[960px] for countdown/bank QR, etc.).

**Recommended transition animation:**
```typescript
// Envelope-to-fullwidth: opacity + translateY
<motion.div
  initial={{ opacity: 0, y: 30 }}
  animate={{ opacity: 1, y: 0 }}
  transition={{ duration: 0.8, ease: 'easeOut' }}
>
  <StickyNav theme={theme} />
  <SharedTemplate invitation={filteredInvitation} theme={theme} />
</motion.div>
```

## Map/Venue Section Strategy

The Stitch designs show a static map image with a bouncing location pin. Options for the map image:

1. **Google Static Maps API** -- requires API key, generates actual map images based on coordinates
2. **Placeholder map image** -- a generic decorative map texture/illustration
3. **OpenStreetMap static image** -- free, no API key, but lower visual quality

**Recommendation:** Use a placeholder map pattern (neutral/grayscale map texture) as the default. The `venueMapUrl` field already exists for the "Chi duong" (Directions) button link. The map section is primarily informational (address text + directions link), not a functional map embed. A decorative placeholder with the bouncing pin icon is exactly what Stitch shows.

**Fallback:** If no venue is configured, hide the entire VenueSection.
